# README_FROM_ME

깃허브 데스크톱 깔아서 올리려다가 깃허브 버전 관리하고 커밋하고 푸시하고 이런것 들이 아직 익숙치 않은데 Raw data 폴더들이 너무 무거워서 자꾸 뻗어버리네요.

스트레스 받아서 그냥 올리게 코드만 올리게 되었습니다, 차차 깃허브 사용법은 배워나가도록 하겠습니다. 개념은 잘 이해했는데 사용이 익숙치가 않네요.

코드를 무작정 짜달라고 하기보단 사전 탐색으로 계획을 치밀히 세우는 것이 중요하다는 것을 알았습니다.

코드에 휘둘려 다니다간 토큰은 토큰대로 쓰고 뭘 고쳐야하는지 AI 스스로도 모르게 되더군요.



## OCR 프로젝트

### [기존 계획]
 번호판 인식 -> 번호판 전처리(크롭 및 회전, 노이즈 제거 등) -> 학습 or 인식 의 순서로 진행합니다.

처음에는 번호판 인식에 OpenCV 자체만을 사용하려고 했었고, 이는 꽤나 잘 먹혀들었습니다. Coutour 및 블러, 꼭짓점 인식하여 회전 등을 먼저 적용하였고
컨투어가 인식된 사진 속 여러 부분들(글자 등)에 번호판과 그 속 숫자들이 갖는 특징(일정 간격으로 숫자 존재, 가로로 평행, aspect ratio 등)을 적용한 탐지 로직을 걸어 최종적으로 그릴, 창문, 범퍼 등을 제외하고 번호판을 인식 및 크롭하도록 하였습니다.

하지만 tesseract 기반의 한국어 OCR이 번호판 특유의 폰트와 형태에 잘 작동하지않아 성공률은 50% 정도 였습니다. 

EasyOCR은 간단하나 검출 및 인식이 꽤나 느리고 성공률도 70% 정도에 그쳤습니다.

이후 성능만 좋다면 다 갖다써도 된다, 텐서플로로 넘어가라는 말씀 들은 뒤, 텐서플로, keras CRNN으로 진행하려다 파이썬 버전 문제에 걸린 김에 비슷한 파이토치 기반 CRNN 모델을 사용하기로 하였습니다.

또한 번호판 검출 자체도 YOLOv8 모델을 사용하기로 계획을 전면 수정하였습니다.


### [새로운 계획]
 Raw-data 점검 -> 번호판 인식(YOLOv8) -> 인식 영역 크롭 및 회전, 패딩, 이미지 크기 균일화 -> 학습 (CRNN 모델) -> 인식률 보완(화이트리스트, 000가0000 규칙, 저화질 고난이도 문제 추가학습)

Raw data가 다소 문제가 있었습니다. 8만장의 번호판 이미지 중 파일명인 번호판 정답이 제대로 작성되지 않은 경우들이 1%가량 존재하였고, 번호판의 일부가 잘리거나 일부에 그늘이 지거나 과도하게 측면, 심각한 저화질 등 학습에 적합하지 않은 데이터가 일부 섞여있었습니다. 정답이 제대로 적혀있지 않은 것들은 검사시에 발견될때마다 지워주고는 있지만, 학습 결과에 영향을 미쳤을 것으로 생각합니다. 또한, 유형이 다르다고 판단된 초록색 구형 번호판이나 지역명이 적힌 노란색 번호판도 제거하였습니다. 학습에 사용할 고품질 경량 raw data를 따로 제작하였고, 세로 픽셀 개수(저화질 사진 제거용), 가로 세로 비율(과하게 측면에서 찍힌 이미지 제거용)을 적용하여 랜덤 추출하여 10000장의 경량 데이터셋을 사용하였습니다. 학습용 데이터는 기존에 만들어놓은 모델을 사용하여 한번 학습 전 점검하는 것이 좋을 것 같았다는 생각이 듭니다. 추가로 대비/ 밝기/ 흔들림 등 을 측정하여 적용하는 방식도 좋았을 것 같습니다.

YOLOv8 같은 경우 AI-hub에서 받은 Raw data Json 파일에 번호판 영역 좌표가 없어서, 직접 labelme 라이브러리로 100장 따주었고 이후 이를 학습시킨 YOLO 모델로 3000장 인식, 이후 인식된 바운딩 박스 수동 검사하며 다시 틀린 부분만 수정해준 뒤 3000장으로 재학습하여 아주 좋은 성능을 보였습니다.

이미지는 초창기에는 현재 cpu 기반 학습속도가 느린것을 감안하여 세로 32px의 저화질로 크롭하였으나, 구글 코랩을 사용하기로 변경한 이후 이미지 화질을 대폭 개선하여 320*96으로 사용하였습니다. 학습 시 loss는 0.01 정도로, validation 성공률은 94% 정도였습니다. 위 과정에서 과적합이나 loss가 개선되지 않는 문제에 빠지지 않도록 learning rate나 가중치를 조절하는 등 파라미터 조절에 관해서 배웠습니다.

이후 두 모델을 사용하여 학습에 사용하지 않은 8만장의 raw data set에서 1000장씩 점검하였을 때 **92~93%**의 성공률을 보였고 ㅓ/ㅏ, ㅗ/ㅜ, ㅇ/ㅎ, ㅁ/ㅂ 등 차이가 크지않은 항목들이 틀리는 경우가 50%, 심각한 저화질 및 측면 사진으로 인해 인식률이 떨어지는 경우 30%, 12가1234를 12가가1234 와 같이 규칙을 지키지 않고 인식하는 경우 10%, 실제론 정답이었던 항목 10% 와 같이 분포하였습니다.

이에 틀리기 쉬운 자모음이 포함된 데이터가 절반 이상 포함된 새로운 학습용 데이터를 크롭하여 기존 모델에 추가학습 시키고, 사전 및 규칙을 추가하여 성공률을 **96%~97%**로 개선하였습니다.


### [추가로 개선할 점]
1. 측면에서 찍은 번호판도 YOLO상에선 잘 인식하므로, 바운딩 박스의 aspect ratio를 픽스하면 올바르게 펼 수 있을 것 같습니다. 또한 그 과정에서 기울어진 글자를 세우는 과정을 추가하면 측면 인식률을 개선할 수 있을것으로 보입니다.

2. 한글 글자가 번호판마다 폰트 크기가 제각각입니다. 그래서 보/모 등을 혼동하는 경우가 잦았던 것 같습니다. 번호판 자체 제작 프로그램으로 가상의 데이터를 생성하여 고난도 문항을 집중 학습시키는 것도 좋은 전략으로 사용할 수 있을 것 같습니다.

3. Raw data 자체부터 차량의 번호판 영역이 어느정도 크롭되어있다보니, 차량 전체를 찍었거나 하는등 번호판이 비교적 작은 이미지에서는 번호판을 인식하지 못하는 현상이 있었음. YOLO가 번호판을 일정 면적 비율을 차지해야하는 것으로 학습한것이 아닌가 하며, 좀 더 다양한 이미지를 사용하여 번호판을 학습시키면 인식률이 올라갈 것 같음.

4. 대용량 데이터를 좀 더 쉽게 다루기 위한 방안을 고려해야 할 것 같음. 현재 코드도 마구잡이로 폴더에 만들고 이미지도 크롭, 회전, 테스트1용, 테스트2용, 원본, 학습용 원본, 실패자료, 등등... 관리가 상당히 어려움.
